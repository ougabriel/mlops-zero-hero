# Preparing Kubernetes Manifests for Model Deployment

Once the Kubernetes cluster is ready, the next step is to **deploy the model container** into the cluster.  
In previous lectures, we:

*   Built a Docker image for the intent classifier model

## 1. Understanding Kubernetes Manifests

When deploying an application to Kubernetes, multiple manifest files need to be created.  
They must be applied in the correct order.

### Correct order of Kubernetes resources:

1.  **Namespace**
2.  **Deployment**
3.  **Service**

Each has a specific purpose explained below.

***

## 2. Step 1: Namespace

A namespace provides **logical isolation** within a Kubernetes cluster.  
Namespaces are essential because:

*   Organizations often share a single cluster across many teams.
*   Namespaces isolate resources for each project or team.
*   RBAC, resource quotas, and limits can be applied per namespace.

For this project, we create a namespace such as:

    intent-namespace

Namespace manifest fields:

*   `apiVersion: v1`
*   `kind: Namespace`
*   `metadata.name: intent-namespace`

After applying the namespace, verify with:

    kubectl get ns
    kubectl get all -n intent-namespace

Initially, there will be no resources inside it.

***

## 3. Step 2: Deployment

You **cannot directly deploy a container** to Kubernetes.  
Containers run **inside Pods**, and Pods are **ephemeral**.

Pods can go down due to:

*   Resource issues
*   CPU throttling
*   Node failures
*   Manual deletion

Because Pods are not reliable, you should never deploy Pods directly.  
Instead, use a **Deployment**.

### Why use a Deployment?

A Deployment manages Pods using a **ReplicaSet controller**.

ReplicaSet responsibilities:

*   Continuously monitor the number of running Pods
*   Automatically create a replacement Pod if one goes down
*   Maintain the desired number of replicas declared in the Deployment manifest

Example:  
If the Deployment specifies `replicas: 2`, the ReplicaSet ensures that two Pods are always running.

### Deployment manifest sections:

1.  `apiVersion`
2.  `kind: Deployment`
3.  `metadata`
4.  `spec`
    *   Specifying number of replicas
    *   Pod template
    *   Pod labels (used later by Services)
    *   Container definition
        *   Image name
        *   Container port

A key part of the Deployment is the **labels**, such as:

    labels:
      app: intent-classifier

These labels are important for the Service to perform service discovery.

***

## 4. Step 3: Service

A Deployment alone is not enough.  
Even with a Deployment, the Pod IP will always change when Pods restart.

For example:

*   Old Pod: `172.16.3.4`
*   New Pod: `172.16.3.5`

If users connect directly to the Pod IP, the application breaks whenever Pods restart.

Kubernetes solves this problem with a **Service**, which is a stable access point.

### What a Service does:

*   Acts as a **proxy** for Pods
*   Tracks Pods dynamically using **label selectors**
*   Provides a stable virtual IP
*   Forwards traffic to healthy Pods

Example selector:

    selector:
      app: intent-classifier

How it works:

*   When new Pods are created, the Service automatically discovers them by label.
*   Clients access the Service, never the Pod IP.
*   Service forwards traffic to whichever Pods are currently running.

### Service types:

*   `ClusterIP` (default)
*   `NodePort`
*   `LoadBalancer`

For this demo, the Service was modified to `NodePort` so it can be accessed via node IP.

***

## 5. End-to-End Kubernetes Resource Flow

1.  **Namespace** is created.
2.  **Deployment** is created inside the namespace.
3.  Deployment creates a **ReplicaSet**.
4.  ReplicaSet creates and maintains **Pods**.
5.  **Service** is created.
6.  Service uses its **selector** to discover Pods.
7.  Users send traffic to the Service.
8.  Service forwards requests to the Pods.
9.  Pods run the model container and return predictions.

## 6. Applying the Manifests

### 6.1 Clone the repository

    git clone <repo-url>
    cd intent-classifier-model
    git checkout origin/kubernetes
    cd 04-k8s-manifests

### 6.2 Apply Namespace manifest

    kubectl apply -f namespace.yaml
    kubectl get ns
    kubectl get all -n intent-namespace

### 6.3 Update Deployment manifest

Replace the image placeholder with the container image from Docker Hub:

Example:

    abhishekf5/intent-classifier-model:v1

Save the file.

Apply the Deployment:

    kubectl apply -f deployment.yaml -n intent-namespace

Verify:

    kubectl get pods -n intent-namespace

Once Pods are in `Running` state, proceed.

### 6.4 Apply Service manifest

Optionally change service type to `NodePort`.

Apply:

    kubectl apply -f service.yaml -n intent-namespace

Verify:

    kubectl get all -n intent-namespace

You should now see:

*   Pods
*   ReplicaSet
*   Deployment
*   Service

This completes the **model deployment** phase.

***

## 7. What Comes Next: Model Serving

Deployment only places the model inside the cluster.

The next task is **Model Serving**, which means making the model accessible externally.

This involves:

*   Deploying an Ingress Controller
*   Creating an Ingress resource
*   Accessing the model via external endpoint

Model serving is covered in the next lecture.
