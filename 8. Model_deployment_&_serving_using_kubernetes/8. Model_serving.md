# Model Serving Overview

## 1. Introduction

In this lecture, we focus on **model serving** in the context of this Kubernetes project.

You already know that model serving is the process of making a trained model **accessible to end users**.  
In simple terms, it is the set of configurations and actions that allow the model to **receive requests and return predictions**.

Although we deployed the model container onto the Kubernetes cluster in the previous lecture, **model deployment alone does not mean the model is publicly accessible**.  
This lecture explains why and introduces the correct, production‑grade method for serving models on Kubernetes: **Ingress**.

***

## 2. Why Deployment Alone Is Not Enough

In the previous lecture:

*   We created a Kubernetes namespace
*   Applied a Deployment (which created Pods)
*   Applied a Service of type **NodePort**

Because the Service is a NodePort, the model can technically be accessed using:

    <NodeIP>:<NodePort>

For example:

*   Node external IP: `X.X.X.X`
*   NodePort assigned by Kubernetes: `31479`

Thus, the following request would work (after opening the NodePort in the security group):

    curl -X POST http://<NodeIP>:31479/predict

This works in the demo environment, and the response will be returned correctly.

However, this is **not appropriate for real-world production environments**.

***

## 3. Why Direct NodePort Access Is Not Used in Real-Time Systems

In real company environments, Kubernetes clusters usually operate inside **secure, disconnected VPC networks**.  
This means:

*   Nodes generally do **not** have public IPs.
*   Even if they do, direct access to worker nodes is usually **blocked for security reasons**.
*   Outbound internet access is restricted; inbound access is tightly controlled.
*   Only internal systems or services within the VPC can access NodePort Services.

As a result:

*   End users (internet clients) cannot directly access NodePort Services.
*   Even internal employees may not directly access the nodes.
*   NodePorts are unsuitable for production traffic routing.

This is why relying on NodePort is not a production‑grade serving strategy.

***

## 4. Production-Grade Model Serving Using Ingress

To make the model accessible in a secure, scalable, and standard way, Kubernetes provides **Ingress**.

### What is Kubernetes Ingress?

Ingress is a Kubernetes resource that defines:

*   How external traffic should enter the cluster
*   Which Services should receive that traffic
*   Routing rules, such as hostnames and paths

However, an Ingress resource on its own does nothing.

### Ingress requires an Ingress Controller

An **Ingress Controller** is a component installed on the cluster that:

*   Watches for Ingress resources
*   Automatically configures a Load Balancer
*   Handles routing traffic to the correct Service
*   For AWS EKS, it often provisions an **AWS Application Load Balancer (ALB)**

Common Ingress Controllers include:

*   NGINX Ingress Controller
*   AWS Load Balancer Controller
*   Traefik
*   HAProxy

In this project, we use an Ingress Controller to expose the model.

***

## 5. How Ingress Enables Model Serving

Once the Ingress Controller is installed:

1.  Kubernetes provisions a **public load balancer** (AWS ALB in this case).
2.  The load balancer is attached to **public and private subnets**.
3.  The load balancer receives **public traffic** from end users.
4.  The Ingress resource defines routing rules.
5.  The Ingress Controller configures the load balancer according to those rules.
6.  When a user accesses the load balancer URL, the request flows as follows:

### Traffic Flow

    User
      ↓
    Load Balancer (public)
      ↓
    Ingress Controller
      ↓
    Kubernetes Service (ClusterIP or NodePort)
      ↓
    Pod
      ↓
    Model Container (inference)

### Key advantages of Ingress

*   Users never need Node IP addresses.
*   Pods can restart or scale; routing is automatically updated.
*   Load balancing and SSL termination can be handled at the ALB/Ingress level.
*   This is the **standard, recommended** approach for model serving in Kubernetes.

***

## 6. Demo vs Real-Time Environments

### Demo Environment (this course)

*   Nodes often have public IPs.
*   You can open the NodePort in security groups.
*   Accessing the model via NodePort works for learning.

### Real Production Environments

*   Nodes are private.
*   NodePort access is blocked.
*   Ingress is mandatory.
*   Ingress Controller provisions a load balancer.
*   Users access the model through the load balancer URL.

Therefore, even though NodePort works for local or demo use, **Ingress is required for real model serving**.

***

## 7. Summary

Model serving is not just exposing a Service.  
It is about providing **secure, scalable, and production‑ready access** to the model.

Key points:

*   NodePorts may work for internal or demo use but are unsuitable for production.
*   Production clusters use private nodes inside a secure VPC.
*   End users cannot access nodes directly, which makes NodePort unusable.
*   Kubernetes Ingress is the correct strategy for serving models.
*   Ingress Controller watches Ingress resources and creates a load balancer.
*   Users access the load balancer URL, not the node or Pod IP.
*   The load balancer forwards traffic to Services, then Pods, then the model.
