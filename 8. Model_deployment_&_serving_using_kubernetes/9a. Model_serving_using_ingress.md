# Ingress Resource Creation and Real-Time Model Serving

*Complete lecture note including all transcript content*

## 1. Introduction

In the previous lecture, we installed and verified the **Traefik Ingress Controller** on the Kubernetes cluster.  
In this lecture, we:

1.  Create the **Ingress resource** for the intent classifier model.
2.  Watch how Traefik responds to the Ingress in real time.
3.  Test model serving using the automatically provisioned **public load balancer**.
4.  Validate the full production-grade model serving workflow.

This is the final step in the model deployment and serving process on Kubernetes.

***

## 2. Preparing to Observe Traefik Logs

To see how Traefik reacts to the Ingress resource, keep Traefik logs streaming in a separate terminal.

1.  Get the Traefik pod name:

<!---->

    kubectl get pods -n traefik

2.  Stream logs:

<!---->

    kubectl logs -f <traefik-pod-name> -n traefik

This will show Traefik events instantly when the Ingress resource is created.

***

## 3. Review of the Ingress Resource

The Ingress file, located in the GitHub repository under:

    06-ingress.yaml

contains the configuration needed to route traffic to the model.

Pull the latest changes:

    git pull

Ensure you are on the `kubernetes` branch.

### 3.1 Ingress Namespace

The Ingress resource must be created inside the same namespace as the Deployment and Service:

    namespace: intent-namespace

### 3.2 Ingress Class Name

The field:

    ingressClassName: traefik

tells Kubernetes that this Ingress resource should be processed **only** by the Traefik controller.

If the cluster contains multiple ingress controllers (for example Traefik and AWS ALB), this ensures the correct controller handles the resource.

### 3.3 Host and Path Rules

Ingress rule configuration:

    host: example.com
    path: /predict
    service: intent-classifier-service

Meaning:

*   Any request to `example.com/predict`
*   Will be forwarded to the Kubernetes Service `intent-classifier-service`
*   Which forwards to Pods running the model container

Since we are not purchasing a real domain for this demonstration, `example.com` is used as a placeholder.

***

## 4. Applying the Ingress Resource

Run:

    kubectl apply -f 06-ingress.yaml

Immediately, Traefik will detect the new resource.  
In the logs, you will see something like:

    Updated Ingress status ... ingress=intent-classifier namespace=intent-namespace

This confirms:

*   Traefik has read the Ingress rules.
*   Traefik has begun provisioning a public load balancer.

***

## 5. Verifying Ingress & Load Balancer Creation

Retrieve the Ingress object:

    kubectl get ingress -n intent-namespace

Expected output includes:

*   The Ingress name
*   Host (`example.com`)
*   The public `ADDRESS` (Load Balancer DNS name)

Example:

    intent-ingress    traefik    example.com    a1b2c3d4e5.elb.amazonaws.com    80    1m

This **public load balancer address** is what external users will access.

In a real company, the team would:

*   Buy a domain
*   Create a DNS A‑record mapping the domain to the load balancer address

Since domain buying is out of scope here, a local DNS override technique is used.

***

## 6. Simulating DNS Using Curl’s `--resolve` Option

The `--resolve` option overrides DNS locally for a single Curl request.

Format:

    curl --resolve <domain>:<port>:<IP> ...

This allows testing without buying a domain.

### 6.1 Get the Load Balancer IP

If tools like `nslookup` or `dig` are unavailable (as in CloudShell), use:

    getent hosts <load-balancer-dns>

This returns an IP such as:

    3.6.197.130

### 6.2 Constructing the Test Command

Example Curl request:

    curl --resolve example.com:80:3.6.197.130 \
        -X POST \
        -H "Content-Type: application/json" \
        -d '{"text":"Hello"}' \
        http://example.com/predict

This forces the request for `example.com` to resolve to the load balancer's IP.

If successful, the output will be the model prediction:

    {"intent":"greeting"}

## 8. Summary

This lecture demonstrated the complete real-time model serving flow:

1.  Keeping Traefik logs open for live monitoring
2.  Applying the Ingress resource
3.  Watching Traefik detect the Ingress and update it
4.  Traefik auto‑provisioning a public load balancer
5.  Using Curl’s `--resolve` to simulate DNS for the demo
6.  Accessing the prediction API through the load balancer
7.  Validating end‑to‑end model serving in Kubernetes

This is the production‑grade pattern used in real organizations:

*   Users access a **Load Balancer**
*   Load Balancer → Traefik Ingress Controller
*   Traefik → Kubernetes Service
*   Service → Pods
*   Pods → Model container
*   Output returned to user
