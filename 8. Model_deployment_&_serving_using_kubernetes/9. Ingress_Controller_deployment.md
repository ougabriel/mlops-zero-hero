# Ingress Controller Deployment (Traefik)

*Complete lecture note including all transcript details*

## 1. Introduction

In this lecture, we install and configure an **Ingress Controller** for model serving on our Kubernetes cluster.  
The chosen Ingress Controller is **Traefik**, a modern and lightweight solution that is rapidly growing in popularity.

The Ingress Controller is required to make the model **publicly accessible** in real‑time environments, because NodePort Services are insufficient for production use.

***

## 2. Why Traefik Ingress Controller?

Historically, the most commonly used controller was **Ingress-NGINX**, maintained by the Kubernetes community.  
However, the community has **announced the retirement of the Ingress-NGINX project**, which means it will no longer be actively developed or maintained.

Because of this:

*   Traefik has become a preferred lightweight alternative.
*   It is easy to install.
*   It supports modern ingress features.
*   It is widely adopted in production environments.

***

## 3. How Ingress Controllers Work

The role of an Ingress Controller is simple:

1.  It is installed **once per cluster** (cluster‑wide installation).
2.  MLOps engineers deploy the controller.
3.  Application teams create **Ingress resources** inside their own namespaces.
4.  The Ingress Controller **watches** these Ingress resources.
5.  Based on each Ingress configuration, it automatically:
    *   Creates a **load balancer**
    *   Updates routing rules
    *   Connects the load balancer to Services inside the namespace

### Example scenario

*   MLOps team installs Traefik controller (cluster-wide).
*   Team A (Intent Classifier model) creates an **Ingress resource** in the `intent` namespace.
*   Traefik reads this resource and creates a **load balancer** dedicated to the project.
*   Team B (Recommendation model) also creates an Ingress resource.
*   Traefik notices it and sets up another load balancer.

Each load balancer can serve **multiple microservices** for a given project via path‑based or host‑based routing.

***

## 4. Installing Traefik Ingress Controller

Traefik is typically installed using **Helm**, which makes installation simple.

If Helm is not installed, install it first.

### 4.1 Check Helm installation

If Helm is missing:

1.  Go to the official Helm installation documentation.
2.  Download the installation script.

Commands used in the lecture:

```bash
curl https://raw.githubusercontent.com/helm/helm/master/scripts/get-helm-3 > get_helm.sh
chmod 700 get_helm.sh
./get_helm.sh
```

Now Helm should be installed.

### 4.2 Add Traefik Helm repository

```bash
helm repo add traefik https://traefik.github.io/charts
```

### 4.3 Install Traefik Controller

```bash
helm install traefik traefik/traefik --namespace traefik --create-namespace
```

This command:

*   Creates the **traefik** namespace
*   Installs the Traefik controller into that namespace
*   Deploys all required Kubernetes objects such as:
    *   Deployment
    *   ReplicaSet
    *   Pod
    *   Service (usually a LoadBalancer type)

***

## 5. What Actually Gets Installed?

Traefik Ingress Controller is **not** a complex external system.  
It is simply:

*   A **Pod** running Traefik controller code
*   A **Service** to expose the controller
*   A **ReplicaSet** that maintains controller availability

This Pod is responsible for watching and processing Ingress resources.

### Verification

```bash
kubectl get all -n traefik
```

You should see:

*   A running Traefik Pod
*   A Service (LoadBalancer or ClusterIP depending on configuration)
*   A ReplicaSet

At this point, the Traefik controller is **running and waiting** for Ingress resources to be created.

***

## 6. How It Works with Your Project

After installation:

1.  Traefik sits idle.
2.  When you create an **Ingress resource** inside the `intent-namespace`, Traefik detects it.
3.  Traefik reads the Ingress YAML rules.
4.  Traefik provisions a **Load Balancer** (AWS LB when using EKS).
5.  The load balancer becomes the **public entry point** for your model.
6.  Requests flow:

<!---->

    User → Load Balancer → Traefik Ingress Controller → Service → Pod → Model Container

You will see all of this in action during the next lecture.

## 7. Summary

Key points from this lecture:

*   Traefik is a lightweight, modern Ingress Controller.
*   It is becoming popular due to the retirement of Ingress-NGINX.
*   Ingress Controllers are installed once per cluster.
*   Application-specific Ingress resources are created inside namespaces.
*   Traefik monitors these resources and automatically provisions load balancers.
*   This is the correct strategy for **production-ready model serving**.
*   We verified that Traefik is installed and running.
