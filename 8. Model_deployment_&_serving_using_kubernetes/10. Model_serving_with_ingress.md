# Model Serving Using Ingress (Traefik)

## 1. Introduction

In the previous lecture, we installed the **Traefik Ingress Controller** on the Kubernetes cluster.  
In this lecture, we create an **Ingress resource**, apply it to Kubernetes, and observe how Traefik reacts to it in real time.

We then test model serving using the newly created public load balancer.

This completes the **model serving** workflow on Kubernetes.

***

## 2. Recap: How Ingress Works

*   The Traefik Ingress Controller monitors the cluster for new **Ingress resources**.
*   When an Ingress resource is created:
    *   Traefik reads the rules defined in that file.
    *   It automatically provisions a **public Load Balancer** (in AWS: an ELB/ALB).
    *   It configures routing:
        *   Incoming traffic to a domain (host)
        *   Matching path (e.g., `/predict`)
        *   Forwarding to a Kubernetes Service
        *   Which then forwards to Pods running the model container

Thus, the flow becomes:

    User → Load Balancer → Traefik Ingress Controller → Kubernetes Service → Pod → Model Container

This is the production-grade method of serving models on Kubernetes.

***

## 3. Preparing the Ingress Resource

We already have the Ingress file in the GitHub repo under:

    06-ingress.yaml

The Ingress file contains:

### 3.1 Namespace

The Ingress resource must be deployed in the same namespace where the model is deployed:

    namespace: intent-namespace

### 3.2 IngressClassName

This determines which controller should process the Ingress:

    ingressClassName: traefik

Why?

*   A single cluster may have multiple ingress controllers.
*   For example: Traefik, AWS ALB Controller, NGINX, etc.
*   The ingressClassName tells Kubernetes:  
    **“This Ingress should be handled by Traefik.”**

### 3.3 Host and Path Rules

Example:

    host: example.com
    path: /predict

Request routing rule:

    If user accesses example.com/predict → send request to intent-classifier-service

The Service then sends the request to the Pods.

Even though we do not own the domain `example.com`, we use it as a placeholder and later override DNS locally using Curl's `--resolve` option.

***

## 4. Applying the Ingress Resource

### 4.1 Apply the Ingress File

    kubectl apply -f 06-ingress.yaml

This creates an Ingress object inside the namespace.

### 4.2 Observe Traefik's Reaction

In another terminal, Traefik logs show:

    Updated Ingress status ... namespace=intent-namespace

This means:

*   Traefik detected the new Ingress resource.
*   It processed the rules.
*   It provisioned a **new public Load Balancer**.
*   It updated the Ingress status with the LB’s public address.

### 4.3 Verify Ingress Creation

    kubectl get ingress -n intent-namespace

Example output:

    NAME               CLASS     HOSTS          ADDRESS                 PORTS   AGE
    intent-ingress     traefik   example.com    a1b2c3d4e5f6.elb.amazonaws.com   80      1m

The `ADDRESS` is the **public Load Balancer URL** created by Traefik.

## 6. Local DNS Override Using Curl (Practical Hack)

Curl provides a `--resolve` option which overrides DNS resolution **only for the current command**.

Syntax:

    curl --resolve <host>:<port>:<IP> ...

Example:

    curl --resolve example.com:80:3.6.197.130 ...

Where:

*   `example.com` → fake domain we chose
*   `80` → port
*   `3.6.197.130` → IP address of the Load Balancer

To get the Load Balancer IP, run:

    kubectl get ingress -n intent-namespace

If IP is not shown, you can resolve it:

    host <load-balancer-dns>

or

    getent hosts <load-balancer-dns>

In the transcript, `getent` worked:

    getent hosts <LB-DNS>

Result (example):

    3.6.197.130

## 7. Testing the Model Serving

Construct the curl command:

    curl --resolve example.com:80:3.6.197.130 \
         -X POST \
         -H "Content-Type: application/json" \
         -d '{"text":"Hello"}' \
         http://example.com/predict

Expected output:

    {"intent":"greeting"}

This confirms:

*   Load Balancer is working.
*   Traefik is routing correctly.
*   Service is forwarding traffic.
*   Pods are running.
*   Model is responding successfully.

You can run this command from:

*   AWS CloudShell
*   Your local terminal

Both will work since the Load Balancer is public.
