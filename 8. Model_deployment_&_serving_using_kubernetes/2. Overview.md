

# Simple Summary

This lecture introduces **model deployment and serving on Kubernetes**, building on what you already learned with **virtual machines**.

Instead of:

*   EC2 instances
*   Auto Scaling Groups
*   Load Balancers

You will now use:

*   **Containers**
*   **Kubernetes (K8s)**
*   **Ingress controllers**

### Key ideas

*   **The same intent classifier model is reused**
*   Deployment strategy changes from **VM‑based** → **container & Kubernetes‑based**
*   All Kubernetes‑specific code, manifests, and notes live in the **`kubernetes` Git branch**
*   Deployment and serving are treated as **two separate phases**

***

### High‑Level Kubernetes Flow

1.  Package the model as a **Docker container**
2.  Push the container to a **container registry**
3.  Deploy the container to a **Kubernetes cluster**
4.  Expose the deployed model using **Ingress**
5.  Perform **model inference** via external API calls

✅ This approach is **cloud‑native, scalable, and production‑grade**

***

# Step‑by‑Step Guide (Kubernetes Labs Overview)

> ⚠️ These are the **steps you will implement across the next lectures**, not all in this one lecture.

***

## Step 0: Switch to the Correct Git Branch (Very Important)

Before doing anything:

```bash
git checkout kubernetes
```

*   `virtual-machines` branch → VM deployment
*   `kubernetes` branch → Kubernetes deployment (this section)

***

## Step 1: Prepare a Dockerfile (Containerization)

**Goal:** Convert the ML model + API into a container

What you will do:

*   Write a `Dockerfile`
*   Copy:
    *   Application code
    *   Model files
    *   WSGI configuration
*   Define:
    *   Base image (Python)
    *   Dependency installation
    *   Startup command (Gunicorn)

✅ Kubernetes can only run **containers**, not raw Python apps

***

## Step 2: Choose a Container Registry (Model Registry)

Since models are packaged as containers, the **model registry = container registry**

Possible options:

*   Amazon ECR (AWS)
*   Google GCR
*   Docker Hub

What you will do:

*   Create a registry repository
*   Authenticate from your local machine or CI pipeline

✅ This is where Kubernetes pulls the model image from

***

## Step 3: Build and Push the Model Image

**Goal:** Make the model image available to Kubernetes

What you will do:

*   Build Docker image from Dockerfile
*   Tag the image
*   Push it to the container registry

✅ This image becomes the deployable model artifact

***

## Step 4: Create / Access a Kubernetes Cluster

**Goal:** Prepare the Kubernetes environment

What you will do:

*   Create a Kubernetes cluster (EKS / GKE / AKS / local)
*   Configure `kubectl` access
*   Apply best practices:
    *   Namespaces for teams
    *   Resource limits
    *   Separation of workloads

✅ One Kubernetes cluster can serve **multiple teams and applications**

***

## Step 5: Write Kubernetes Deployment Manifest

**Goal:** Deploy the model container to Kubernetes

What you will do:

*   Create a `Deployment` YAML
*   Define:
    *   Image name
    *   Replica count
    *   CPU & memory requirements
    *   Container ports

✅ This is where **model deployment** officially happens

***

## Step 6: Create Supporting Kubernetes Resources (If Needed)

Depending on the setup, you may create:

*   **ConfigMaps** → non‑secret configs
*   **Secrets** → credentials, tokens
*   Volume mounts (if required)

✅ Keeps configuration clean and secure

***

## Step 7: Deploy the Model to Kubernetes

**Goal:** Run the model inside the cluster

What you will do:

```bash
kubectl apply -f deployment.yaml
```

*   Pods start running
*   Kubernetes handles restarts and scaling

✅ Model is now **running**, but not yet accessible externally

***

## Step 8: Install an Ingress Controller (Model Serving Setup)

**Problem:**  
Users outside Kubernetes cannot access the model yet

**Solution:**  
Ingress controller (e.g. NGINX Ingress)

What you will do:

*   Deploy ingress controller to the cluster
*   Configure cluster networking

✅ This replaces VM‑based load balancers

***

## Step 9: Create Kubernetes Ingress Resource

**Goal:** Expose the model API externally

What you will do:

*   Create an `Ingress` YAML
*   Map:
    *   URL / domain
    *   Service
    *   Internal pod port

✅ External traffic → Ingress → Model service → Pod

***

## Step 10: Test Model Inference (End‑to‑End)

**Goal:** Validate complete deployment & serving

What you will do:

*   Send HTTP request from outside the cluster
*   Hit the Ingress endpoint
*   Receive model prediction response

✅ Model inference is now live on Kubernetes

***
