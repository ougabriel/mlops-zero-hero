
# Simple Summary

This lecture explains **how to package the intent classifier model as a Docker container**, which is a **mandatory step before deploying the model to Kubernetes**.

Instead of running:

*   Python scripts directly (VM‑style)
*   User‑data shell scripts

You now:

*   Build a **Docker image**
*   Run the model API inside a **container**
*   Use **Gunicorn (WSGI)** for production‑grade serving

### Key Concepts Covered

*   Choosing a **Python slim base image**
*   Optimizing Docker layers for **faster builds**
*   Installing dependencies efficiently
*   Training the model **inside the container**
*   Exposing the API port
*   Running the API using **Gunicorn** instead of Flask dev server

✅ The result is a **reusable, portable model image** that Kubernetes can deploy at scale.

***

# Step‑by‑Step Lab Guide: Preparing the Dockerfile

> 📌 **Important**
>
> *   Switch to the **`kubernetes` branch**
> *   A reference Dockerfile already exists there

```bash
git checkout kubernetes
```

***

## Step 1: Understand the Container Requirements

Your Docker image must:

*   Have Python installed
*   Contain application + model code
*   Install all dependencies
*   Train the model
*   Start the API using Gunicorn
*   Listen on **port 6000**

***

## Step 2: Create the Dockerfile

Create a new file named **Dockerfile** (no extension):

```bash
touch Dockerfile
```

***

## Step 3: Choose a Python Base Image

```dockerfile
FROM python:3.10-slim
```

✅ Why `python:slim`?

*   Python comes pre‑installed
*   Smaller image size
*   Fewer OS vulnerabilities

***

## Step 4: Set Python Environment Variables (Optional but Best Practice)

```dockerfile
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1
```

**Why this matters**

*   Prevents `.pyc` files
*   Ensures real‑time logs (stdout / stderr)
*   Cleaner, safer containers

***

## Step 5: Set the Working Directory

```dockerfile
WORKDIR /app
```

✅ All commands now run inside `/app`

***

## Step 6: Copy Dependencies First (Layer Optimization)

```dockerfile
COPY requirements.txt .
```

✅ This is done **before copying source code** to leverage Docker layer caching.

***

## Step 7: Install System & Python Dependencies (Single RUN Layer)

```dockerfile
RUN apt-get update && \
    apt-get install -y gcc && \
    pip install --no-cache-dir -r requirements.txt && \
    apt-get remove -y gcc && \
    apt-get autoremove -y && \
    apt-get clean
```

### Why this approach is important

*   Fewer Docker layers → faster builds
*   Smaller final image
*   Dependencies are cached unless `requirements.txt` changes

***

## Step 8: Copy the Application Source Code

```dockerfile
COPY . .
```

✅ All model, API, and config files are now inside the image

***

## Step 9: Train the Model During Image Build

```dockerfile
RUN python3 model/train.py
```

✅ This creates:

*   `artifacts/intent_model.pkl`

📌 The API (`app.py`) later loads this model from the artifacts directory.

***

## Step 10: Expose the API Port

```dockerfile
EXPOSE 6000
```

⚠️ Note:

*   `EXPOSE` is **metadata only**
*   Actual exposure happens via Docker/Kubernetes networking

***

## Step 11: Run the API Using Gunicorn (WSGI)

```dockerfile
CMD ["gunicorn", "--workers", "4", "--bind", "0.0.0.0:6000", "app:app"]
```

### Why Gunicorn?

*   Supports concurrency
*   Production‑grade WSGI server
*   Same logic you used earlier on VMs

***

## Final Dockerfile (Clean View)

```dockerfile
FROM python:3.10-slim

ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

WORKDIR /app

COPY requirements.txt .
RUN apt-get update && \
    apt-get install -y gcc && \
    pip install --no-cache-dir -r requirements.txt && \
    apt-get remove -y gcc && \
    apt-get autoremove -y && \
    apt-get clean

COPY . .
RUN python3 model/train.py

EXPOSE 6000

CMD ["gunicorn", "--workers", "4", "--bind", "0.0.0.0:6000", "app:app"]
```
