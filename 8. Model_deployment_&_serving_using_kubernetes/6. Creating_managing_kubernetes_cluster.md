# ✅ **2. Two Ways to Create a Kubernetes Cluster**

Kubernetes clusters can be created using:

***

## ✅ **A) Local Kubernetes Cluster (Free)**

Used for:

*   Demos
*   Proof of concept
*   Learning
*   Zero‑cost training

Common tools:

### ✅ **kind** (Kubernetes‑in‑Docker) – *Recommended*

*   Extremely lightweight
*   Entire Kubernetes cluster runs inside Docker
*   Setup takes **< 2 minutes**
*   Perfect for student laptops
*   Good for ALL demos in this course

### ✅ Minikube

*   VM‑based
*   Heavier than kind
*   Supports addons like metrics-server, ingress, dashboard

### ✅ MicroK8s

*   Lightweight
*   Runs natively on Linux/macOS
*   Created by Canonical (Ubuntu)

***

## ✅ **Limitations of Local Clusters**

These distributions technically support **Ingress**, but:

### ❌ They CANNOT expose applications publicly on the internet.

Only people inside your machine/network can reach the model.

Local clusters require `kubectl port-forward` to allow temporary external access:

```bash
kubectl port-forward svc/my-service 8080:80
```

⚠️ **This is NOT meant for production.**

***

## ✅ **Advantages of Local Clusters**

*   Completely **free**
*   No need for cloud setup
*   Easy for learning, testing, experimenting
*   Full Kubernetes API available

***

# ✅ **3. kind (Kubernetes in Docker) — Quick Demo**

kind is recommended for this course.

### ✅ Install kind (example: macOS)

```bash
brew install kind
```

### ✅ Create a cluster

```bash
kind create cluster --name demo-cluster
```

A full Kubernetes cluster is created in a Docker container within **60–120 seconds**.

### ✅ Verify

```bash
kubectl get nodes
kubectl cluster-info
```

You can:

*   Deploy applications
*   Create Services
*   Use Ingress (local-only)

But again, ingress is *not public*.

# ✅ **6. Preparing the Kubernetes Cluster (EKS)**


If eksctl is missing, install it from the official AWS guide using the Linux commands shown in the lecture.

***

# ✅ **7. Creating an EKS Cluster Using eksctl**

Command used:

```bash
eksctl create cluster \
  --name demo-cluster \
  --region ap-south-1 \
  --version 1.32 \
  --nodegroup-name demo-nodes \
  --node-type t3.medium \
  --nodes-min 1 \
  --nodes-max 3
```

Notes:

*   `t3.medium` chosen to **minimize cost**
*   Min nodes = 1
*   Max nodes = 3
*   The Intent Classifier model is intentionally lightweight to keep costs low

Cluster creation takes **15–20 minutes**.

# ✅ **9. Configure kubectl to Use the New Cluster**

After creation:

```bash
aws eks update-kubeconfig --name demo-cluster --region ap-south-1
```

Verify:

```bash

```

    demo-cluster

Now:

```bash

```

If nodes appear → EKS is ready.

# ✅ **10. CLUSTER MANAGEMENT IN REAL COMPANIES**

This is **very important for interviews**.

Companies typically have **three main environments**:

1.  **Development**
2.  **Staging / QA / UAT**
3.  **Production**

Each environment has **one Kubernetes cluster**.

Not dozens — **ONE cluster per environment**.

***

# ✅ **11. How Clusters Are Shared Across Teams**

A large organization (like an e‑commerce company) may have teams:

*   Payments Team
*   Shipping/Logistics Team
*   Login/Authentication Team
*   Catalog Team
*   Fraud Detection Team
*   MLOps Team

All need Kubernetes.

Instead of giving each team a cluster (too expensive), DevOps/MLOps engineers do:

✅ **One cluster**  
✅ **Multiple namespaces**

Example:

    payments-namespace
    orders-namespace
    shipping-namespace
    auth-namespace
    catalog-namespace
    mlops-namespace

***

# ✅ **12. Namespace Isolation**

Namespaces are **logical isolation**, NOT physical.

Nodes = physical  
Namespaces = virtual slices

Each team gets:

*   Its own namespace
*   Its own ServiceAccounts
*   RBAC rules
*   Resource quotas

***

# ✅ **13. Resource Quotas (Very Important)**

DevOps/MLOps engineers assign:

```yaml
cpu: 4
memory: 4Gi
```

Meaning:

*   All microservices in the namespace **must fit within 4 CPU / 4GB RAM total**.

If they exceed:

✅ Pods go into **OOMKilled**  
✅ CPU throttling  
✅ Kubernetes prevents overconsumption

This keeps:

*   Costs controlled
*   Teams isolated
*   Production stable

***

# ✅ **14. Pod-Level Limits**

Teams then assign Requests & Limits per pod:

Example:

*   Pod A → 1 CPU
*   Pod B → 500m CPU
*   Pod C → 200m CPU

They must manage usage within their **namespace quota**.

***

# ✅ **15. Perfect Interview Answer (Memorize This)**

> “We maintain **one Kubernetes cluster per environment** (dev, staging, prod).  
> Each team gets its own **namespace**, isolated using RBAC.  
> We apply **ResourceQuota** on each namespace to restrict CPU/memory usage.  
> Teams use **resource requests and limits** in pod specs to ensure stable resource allocation.”

This is exactly what interviewers expect.
