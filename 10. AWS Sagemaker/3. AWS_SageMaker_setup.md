# SageMaker AI “Real-Time Setup” (Org-Grade) — Explained Clearly

The lecture boils SageMaker AI onboarding into **3 core steps**:

1.  **Create Domains (team/project workspaces) + baseline IAM roles**
2.  **Create User Profiles (people) + app-level configuration**
3.  **Enforce isolation using ABAC** so each IAM identity can only open **their own** Studio profile

If you answer *only* “I used AWS CLI to create a domain,” you’ll sound junior. If you explain these **three steps with ABAC**, you’ll sound like someone who has done it in production.

***

## 0) The Business Scenario (Why domains exist)

Example org: `example.com` wants two ML products:

*   **User traffic forecasting model** (internal reliability/capacity planning)
*   **Cost prediction model** (customer-facing predictions)

There are **two teams** building them → MLOps decides to map **team/project boundaries** to domains.

> **Key point:** Domain boundaries are an *organizational control plane decision* (teams, data boundaries, network needs, cost allocation), not just “whatever is convenient.”

***

# ✅ Step 1 — Create Domains (and attach the right execution role)

### What happens here

MLOps sets up **Domain(s)**, and while doing it, they also create/attach **IAM execution roles** because:

*   Studio apps (JupyterLab, Pipelines, Processing jobs, Training jobs, etc.)
*   need permission to create and access underlying AWS resources  
    (EC2, ECR, S3, CloudWatch Logs, KMS, etc.)

### Real-world domain decisions

Common strategies:

*   **One domain per team** (e.g., `traffic-ml`, `pricing-ml`)
*   **One domain per environment** (e.g., `ml-dev`, `ml-prod`)
*   **One domain for org** + strict ABAC on profiles and resources (common in large enterprises)

The lecture’s approach = **domain per model team**, which is valid and easy to reason about early on.

***

## CLI shape (high-level)

MLOps typically prefers **AWS CLI / IaC** because:

*   repeatable
*   auditable
*   less human error
*   supports multi-account, multi-env rollouts

Domain creation is usually done with:

*   `aws sagemaker create-domain ...`

> You’ll supply: VPC/subnets, auth mode, default execution role, S3 settings, etc.

***

# ✅ Step 2 — Create User Profiles (and scope the Studio experience)

### What happens here

Within each domain, MLOps creates **user profiles** for team members. Example in transcript:

Traffic model team has:

*   2 Data Scientists: **Alice**, **Ram**
*   1 ML Engineer: **XYZ**
*   1 MLOps Engineer: **ABC**

So MLOps creates **4 user profiles** in that domain.

### App access & UX configuration

The lecture mentions “enable Jupyter for DS, pipelines/MLflow for MLOps,” which is the right concept, but here’s the practical nuance:

*   Some UI toggles are **visibility/UX**
*   Real enforcement should be **IAM policy + role separation**

✅ Good pattern:

*   DS execution role: training + limited S3 paths
*   ML engineer role: pipelines + registry + deployment tooling
*   MLOps role: broader CI/CD + orchestration permissions

***

# ✅ Step 3 — ABAC (the most important part)

This is the “production safety” step.

## The problem ABAC solves

If you stop after step 1 & 2, then:

*   Alice logs into AWS console
*   Alice can potentially open **Ram’s profile** or **MLOps profile**
*   She could see/modify things she shouldn’t (pipelines, notebooks, experiments)

Even if it’s “unlikely,” in security you design for *possible*, not *probable*.

***

## ABAC in one sentence

**Attribute-Based Access Control** uses **tags** (attributes) on:

*   the **resource** (SageMaker user profile)
*   and/or the **principal** (IAM identity)

…and then enforces access using IAM **Condition** logic like:

> “Allow access only if resource tag Owner == principal tag Owner”

***

# 🔐 Practical ABAC Pattern for SageMaker Studio User Profiles

## 1) Tag each user profile

When you create a user profile, tag it for ownership, e.g.:

*   `Owner = alice`
*   `Team = traffic-ml`
*   `CostCenter = 1234` (optional)

## 2) Tag the IAM principal (user/role)

Tag the IAM user/role for Alice with:

*   `Owner = alice`

## 3) IAM policy condition ties them together

Then the policy says: Alice can only “open” / “presign” Studio URLs for profiles whose `Owner` matches her principal tag.

***

## Example IAM policy snippet (ABAC)

> **Note:** Condition keys vary by service/action; the pattern below is the important part. In practice, you test the exact condition keys supported for the specific SageMaker actions you’re controlling.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowStudioOnlyForMatchingOwnerTag",
      "Effect": "Allow",
      "Action": [
        "sagemaker:DescribeUserProfile",
        "sagemaker:ListUserProfiles",
        "sagemaker:CreatePresignedDomainUrl"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:ResourceTag/Owner": "${aws:PrincipalTag/Owner}"
        }
      }
    }
  ]
}
```

### Why this works

*   User profile resources carry `Owner=alice`, `Owner=ram`, etc.
*   Principal (IAM user/role) carries `Owner=alice`
*   Alice can only perform Studio access actions when the target resource has matching `Owner`

# 🧩 How this maps to the lecture’s “3 steps” (interview answer)

Here’s a strong interview response structure you can use almost verbatim:

### ✅ “How do you set up SageMaker AI in a real organization?”

1.  **Domain setup**: create domains aligned to team/project boundaries using CLI/IaC; configure VPC/subnets, storage, Studio apps, and attach baseline execution roles.
2.  **User profile provisioning**: create profiles per persona (DS/ML Eng/MLOps), configure their environment and assign appropriate execution roles / access patterns.
3.  **ABAC enforcement**: tag user profiles and IAM principals; apply IAM policies with tag-based conditions so each user can only open their own Studio profile, preventing cross-profile access and accidental or malicious changes.

That’s exactly what the lecture is trying to teach: **setup isn’t complete without ABAC**.

# 💡 Extra “production” tips (DevOps/MLOps-grade)

Since you’re a Senior DevOps Engineer, these are the things you’ll care about in a real rollout:

## Networking

*   Decide: public internet allowed vs private-only
*   If private-only: plan for **VPC endpoints** (S3, ECR, Logs, STS, etc.)

## Logging & audit

*   CloudTrail for SageMaker API calls
*   CloudWatch log groups per workload
*   S3 bucket policies to enforce encryption + limited prefixes per team

## Role separation

*   Studio execution role ≠ pipeline role ≠ deployment role
*   Keep blast radius small and permissions least-privilege

## Environment strategy

*   Separate domains or separate AWS accounts for dev/test/prod (very common)
