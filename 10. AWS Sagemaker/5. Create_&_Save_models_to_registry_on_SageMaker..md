# Summary: Creating, Storing, and Registering Models in SageMaker AI (Practical Walkthrough)

This lecture explains how data scientists use SageMaker Studio to create machine learning models and store them in a model registry, and how MLOps engineers set up the environment needed for this workflow. It focuses on two main stages of the full ML lifecycle:

1.  Creating the model
2.  Storing the model version in a model registry (S3 bucket)

Deployment and inference stages will be covered later.

***

## 1. Overview of the Full Workflow

SageMaker AI involves four major stages:

1.  Create the model
2.  Store the model version in a model registry
3.  Deploy the model
4.  Serve the deployed model for inference

This lecture focuses only on steps 1 and 2.

***

## 2. What is Being Built

The lecture demonstrates the workflow used by data scientists:

*   Use SageMaker Studio
*   Launch JupyterLab
*   Use a notebook instance to write Python code
*   Train a model (example: DecisionTreeClassifier on the Iris dataset)
*   Save the trained model
*   Upload the saved model to an S3 bucket (used as the model registry)

As an MLOps engineer, you need to understand this workflow because you depend on the model artifacts being pushed to S3 before you can deploy them.

***

## 3. Step-by-Step Practical Implementation

### Step 1: Create an S3 Bucket (Model Registry)

*   An S3 bucket is created manually via AWS Console.
*   This bucket will store model artifacts (.pkl files etc.).
*   A dedicated folder (for example, `model-artifacts/`) holds the saved models.

***

### Step 2: Create SageMaker Domain Infrastructure

This includes IAM roles, policies, networking, and the domain itself.

#### 2.1 Create the role trust policy (trust.json)

*   Defines `sagemaker.amazonaws.com` as the principal
*   Allows SageMaker to assume the role

#### 2.2 Create the execution role

*   Run `aws iam create-role` with the trust policy
*   If the role already exists, the CLI shows an error (safe to ignore)

#### 2.3 Attach required permissions

Two policies must be attached:

1.  AmazonSageMakerFullAccess
2.  AmazonS3FullAccess

S3 access is required because the model will be uploaded to the S3 model registry.

#### 2.4 Gather network configuration

*   Retrieve default VPC ID
*   Retrieve subnet IDs
*   At least two subnets are required for Studio domain creation
*   Use the default VPC or a fully configured custom VPC

#### 2.5 Create the SageMaker Domain

Run `aws sagemaker create-domain` with:

*   Domain name
*   Authentication mode set to IAM
*   VPC ID and subnet IDs
*   Execution role ARN
*   Network access type set to VPC\_ONLY

After running the command, the domain appears in "Preparing" or "Pending" state.  
Wait until it becomes "InService" before proceeding.

***

### Step 3: Create a User Profile

*   Use `aws sagemaker create-user-profile`
*   Provide the domain ID
*   Provide a user profile name

No ABAC or IAM user is created in this lecture because the focus is model creation.  
The admin logs into SageMaker Studio directly via the console.

***

## 4. Using SageMaker Studio to Create and Register a Model

### 4.1 Launching JupyterLab

*   Open Studio using the created user profile
*   Start a compute instance
*   Launch JupyterLab
*   Open a new `.ipynb` notebook

### 4.2 Writing the Model Code

Notebook contains three sections:

#### Block 1: Import dependencies

*   numpy, pandas, scikit-learn, joblib
*   Load Iris dataset

#### Block 2: Train and save the model

*   Train DecisionTreeClassifier
*   Save as `iris_model.pkl` using joblib

#### Block 3: Upload model to S3

*   Use boto3 S3 client
*   Upload the saved `.pkl` file to the S3 bucket
*   This is the model registry step

Execution is done block-by-block.  
Important note: Only execute a block when the kernel state returns to idle.

***

## 5. Verifying Model Storage

After running the final block:

*   Navigate to the S3 bucket
*   Verify the presence of `iris_model.pkl` inside the `model-artifacts` folder

This confirms that the model is created and stored in the model registry.

***

## 6. Key Understanding for MLOps Engineers

Even though model creation is the responsibility of data scientists:

*   MLOps engineers must ensure that the environment (domain, IAM roles, S3 permissions, networking) works correctly.
*   The only critical output MLOps engineers need is the trained model artifact stored in S3.
*   Once the model is available in the registry, MLOps can proceed with deployment and inference setup.
