

# Simple Summary

This lecture explains **why and how to write a user data script** for deploying an ML model when using **Auto Scaling Groups (ASG)**.

### Why user data scripts are needed

*   In real systems, **user traffic is unpredictable**
*   A **single VM** cannot handle sudden spikes (e.g., from 100 → 1000 requests)
*   Running **many VMs all the time** is expensive
*   **Auto Scaling Groups** solve this by:
    *   Adding VMs when load increases
    *   Removing VMs when traffic drops
    *   Saving cloud costs

### The missing piece: Configuration

Auto Scaling Groups **only create VMs**, not configured environments.

Without configuration:

*   No model
*   No API
*   No dependencies
*   Load balancer sends traffic → instance returns errors

✅ **User data scripts solve this problem**

***

### What is a User Data Script?

*   A **shell script** executed automatically **when a VM is created**
*   Defined inside a **Launch Template**
*   Used to:
    *   Install system dependencies
    *   Clone code
    *   Set up Python environment
    *   Train model
    *   Start API using Gunicorn (WSGI)
    *   Start Nginx as a reverse proxy
    *   Register services with `systemd`

By the time the VM is ready:
✅ The API is live  
✅ Services auto‑start even after reboot  
✅ Load balancer can safely route traffic

***

### Repository Structure Strategy

*   **main branch** → common code (train.py, app.py, wsgi.py)
*   **virtual-machines branch** → VM‑specific deployment (user‑data script, AWS steps)
*   Future branches:
    *   `kubernetes`
    *   `kserve`

This makes it easy to switch deployment strategies.

# Step‑by‑Step Lab Guide

## Part 1: Writing the User Data Script

### Step 1: Create the Script File

Create a new file:

```bash
user-data.sh
```

Start with:

```bash
#!/bin/bash
set -e
```

***

### Step 2: Create Application Directory

```bash
APP_DIR="/opt/intent-app"
mkdir -p $APP_DIR
cd $APP_DIR
```

***

### Step 3: Update System & Install Dependencies

```bash
apt update -y
apt install -y git python3 python3-venv python3-pip nginx
```

✅ Ensures compatibility across Ubuntu/Debian systems

***

### Step 4: Clone the Repository

```bash
git clone <INTENT-CLASSIFIER-REPO-URL> .
```

***

### Step 5: Set Up Python Virtual Environment

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
```

***

### Step 6: Install Python Dependencies

```bash
pip install -r requirements.txt
```

***

### Step 7: Train the Model

```bash
python3 model/train.py
```

✅ Creates model artifacts needed by the API

***

### Step 8: Configure Gunicorn as a systemd Service

Create:

```bash
/etc/systemd/system/intent-gunicorn.service
```

Example configuration:

```ini
[Unit]
Description=Intent Classifier Gunicorn Service
After=network.target

[Service]
User=ubuntu
Group=ubuntu
WorkingDirectory=/opt/intent-app
ExecStart=/opt/intent-app/.venv/bin/gunicorn --workers 3 --bind 0.0.0.0:6000 wsgi:application
Restart=always

[Install]
WantedBy=multi-user.target
```

***

### Step 9: Configure Nginx as Reverse Proxy

Create:

```bash
/etc/nginx/conf.d/intent-app.conf
```

```nginx
server {
    listen 80;

    location / {
        proxy_pass http://127.0.0.1:6000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

✅ Nginx forwards traffic from port **80 → 6000**

***

### Step 10: Enable and Start Services

```bash
systemctl daemon-reload
systemctl enable intent-gunicorn
systemctl start intent-gunicorn

systemctl enable nginx
systemctl restart nginx
```

✅ Services auto‑start on reboot

***

## Part 2: Testing the Script (Before ASG)

### Step 11: Create a Fresh Ubuntu EC2 Instance

*   No configuration
*   Allow HTTP (port 80)
*   t3.micro is sufficient

***

### Step 12: Copy & Run the Script

```bash
chmod 777 test.sh
sudo ./test.sh
```

✅ Run as `sudo` because ASG executes scripts as root

***

### Step 13: Verify Gunicorn API

```bash
curl -X POST http://127.0.0.1:6000/predict \
-H "Content-Type: application/json" \
-d '{"text":"Hey good morning"}'
```

Expected output:

```json
{"intent":"greeting"}
```

***

### Step 14: Verify systemd Service

```bash
systemctl status intent-gunicorn
```

✅ Confirms API is running as a Linux service

***

### Step 15: Verify Nginx Reverse Proxy

```bash
curl -X POST http://localhost:80/predict \
-H "Content-Type: application/json" \
-d '{"text":"Hey good morning"}'
```

✅ Response confirms:

*   Request → Nginx
*   Nginx → Gunicorn
*   Gunicorn → Model

***

### Step 16: Test via Public IP

```bash
curl -X POST http://<EC2-PUBLIC-IP>/predict \
-H "Content-Type: application/json" \
-d '{"text":"Hello"}'
```

✅ Confirms production‑style access

***

# Key Takeaways

*   **Auto Scaling Groups need configuration**
*   **User Data Scripts turn empty VMs into ready‑to‑serve ML servers**
*   Script installs:
    *   Code
    *   Model
    *   API
    *   WSGI
    *   Nginx
*   Services are **fault‑tolerant and restart‑safe**
*   Script must be **tested manually before ASG**
