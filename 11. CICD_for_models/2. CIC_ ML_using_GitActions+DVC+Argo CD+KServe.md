# Summary: End‑to‑End MLOps Implementation for Customer Churn Model

This lecture demonstrates a full real‑world MLOps implementation for a customer churn prediction model. Unlike earlier lectures that used Kubeflow Pipelines, this project uses an **open‑source MLOps stack** consisting of:

*   **DVC** for data versioning
*   **KServe** for scalable model serving
*   **Kubernetes** as the deployment platform
*   **GitHub Actions** for CI (automation of training and packaging)
*   **Argo CD** for CD (continuous delivery to Kubernetes)

This stack mirrors the types of architectures used in real industry-grade MLOps teams, and the delivered project can be used directly on a resume.

***

# 1. Real‑World Use Case: Customer Churn Prediction

Industries like telecom, retail, e‑commerce, and quick‑commerce rely heavily on churn prediction.  
Example scenario: telecom providers (e.g., Verizon or AT\&T).

### Why churn prediction matters

*   Customers today switch providers easily.
*   Support and retention teams need to identify customers likely to leave.
*   ML models can predict the probability of churn based on customer behavior and history.
*   Once identified, support teams offer targeted discounts/promotions to retain the customer.

### Dataset features typically used

*   Age
*   Tenure (years or months with company)
*   Monthly & yearly charges
*   Number of support cases
*   Historical churn data

Example patterns:

*   Younger customers churn more frequently.
*   Customers with long tenure churn less.
*   Many support issues indicate high churn probability.

# 3. Understanding the Project Repository

The GitHub repo for the project has two branches:

*   **main branch** → what data scientists and ML engineers produce:
    *   `train.py`: trains RandomForest model and saves `churn_model.pkl`
    *   `api.py`: FastAPI inference endpoint `/predict`
    *   `generate_data.py`: simulates dataset for training
    *   `data/` & `models/` directories
    *   Documentation

*   **ci-cd branch** → contains full MLOps implementation:
    *   DVC configuration
    *   GitHub Actions workflow
    *   KServe inference manifest
    *   Argo CD application config
    *   Model automation logic

The student uses the `main` branch as the “starting point” and implements full MLOps in the `ci-cd` branch.

# 5. MLOps Component 1: DVC (Data Version Control)

Purpose: version the dataset using S3 as the remote storage.

Process:

1.  Install DVC + DVC S3 dependencies
2.  Run `dvc init`
3.  Create S3 bucket for DVC storage
4.  Add remote:  
    `dvc remote add -d s3remote s3://bucket-name`
5.  Add dataset to DVC:  
    `dvc add data/churn_data.csv`
6.  Push dataset to S3:  
    `dvc push`

DVC creates `.dvc` metadata files and maintains checksums so all team members use the same version of the dataset.

***

# 6. MLOps Component 2: Model Registry via S3

*   The trained model (`churn_model.pkl`) is uploaded to a separate folder in the same S3 bucket.
*   This serves as the **model registry**.
*   New model versions overwrite or co-exist depending on CI configuration.

***

# 7. MLOps Component 3: Kubernetes + KServe

### Steps:

1.  Create a Kubernetes cluster using **Kind**

2.  Install **cert-manager**

3.  Install **KServe CRDs and controller**

4.  Create namespace `ml`

5.  Create **service account** with AWS credentials so KServe can pull model from private S3 bucket

6.  Create **inference.yaml** describing:
    *   Model storage (S3 path)
    *   Framework type (sklearn)
    *   Service account to use
    *   Predictor configuration

7.  Deploy:
    `kubectl apply -f inference.yaml`

### Result:

KServe creates:

*   Deployment
*   Pod
*   Service
*   Autoscaler (HPA)  
    All automatically.

The model runs inside Kubernetes.

***

# 8. MLOps Component 4: GitHub Actions (Automated Training & Packaging)

GitHub Actions workflow automates:

1.  Checkout code
2.  Set up Python
3.  Install dependencies
4.  Generate new dataset
5.  Train model
6.  Configure AWS credentials
7.  Upload new model to S3
8.  Update `inference.yaml` with new model path
9.  Commit updated file back to `ci-cd` branch

This ensures every new commit → new model version → new updated inference manifest → auto‑deployment.

***

# 9. MLOps Component 5: Argo CD (Continuous Delivery)

Steps:

1.  Install Argo CD in the same cluster
2.  Port-forward the Argo server UI
3.  Login using initial admin password
4.  Create an Argo CD application:
    *   Points to GitHub repo
    *   Points to `ci-cd` branch
    *   Points to `k8s/` directory (manifests)
    *   Enabled auto-sync

Whenever GitHub Actions updates the inference YAML:

*   Argo CD detects the change
*   Syncs the new state
*   Deploys the new model automatically

This forms the complete CD pipeline.

***

# 10. Final End‑to‑End Flow

1.  **Code pushed to ci-cd branch**
2.  **GitHub Actions** runs:
    *   Generate data → train model → upload model → update inference YAML
3.  **Argo CD** detects updated inference file
4.  **Argo CD** deploys new version of model to Kubernetes using KServe
5.  **KServe** automatically scales, manages the runtime, and exposes inference endpoint
6.  Model inference is accessible, tested via curl

This completes a full MLOps workflow:

*   Data versioning
*   Model training
*   Model versioning
*   Automated deployment
*   GitOps-based continuous delivery
