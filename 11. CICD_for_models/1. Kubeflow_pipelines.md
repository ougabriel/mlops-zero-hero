# Summary of the Kubeflow Lecture

This lecture introduces Kubeflow, explains why it is used in MLOps, breaks down its main components, and demonstrates how to install Kubeflow Pipelines and create both a simple and a real‑world pipeline. The goal is to show how Kubeflow automates the machine learning lifecycle and reduces operational effort for ML engineers.

***

# 1. What Kubeflow Is

Kubeflow is a comprehensive, Kubernetes‑native MLOps platform designed to automate and manage the complete machine learning lifecycle. It is not a single tool but a collection of components that work together to support data loading, feature engineering, training, tuning, deployment, and monitoring.

Kubeflow’s objective is to make ML workflows scalable, reproducible, automated, and easy to manage on Kubernetes.

***

# 2. Why Kubeflow Is Needed

Machine learning lifecycle stages are highly iterative and operationally heavy:

*   Problem definition
*   Data collection
*   Data analysis
*   Feature engineering
*   Model training
*   Model evaluation
*   Model deployment
*   Model monitoring
*   Model retraining when performance drops

In practice, many models never reach production because the ongoing manual work (retraining, evaluation, versioning, deployment) is too difficult to manage. MLOps solves this by automating the lifecycle, just as DevOps automated traditional software development.

Kubeflow is an MLOps platform that automates ML workflows using Kubernetes.

***

# 3. Components of Kubeflow

Kubeflow is composed of multiple sub‑projects, each targeting a specific stage in the ML lifecycle:

1.  **Kubeflow Pipelines (KFP)**
    *   Pipeline orchestration
    *   Data loading, preprocessing, training, evaluation
    *   Runs each step as containers/pods on Kubernetes

2.  **Kubeflow Notebooks (Jupyter Notebooks)**
    *   Interactive development environment for data scientists

3.  **Katib**
    *   Hyperparameter tuning

4.  **KFServing (KServe)**
    *   Model deployment and serving

5.  **Model Registry**
    *   Versioning and storing model artifacts

6.  **Training Operator**
    *   Supports distributed training jobs

7.  **Kubeflow Dashboard**
    *   Unified UI to access all Kubeflow components

Kubeflow can be installed fully or component‑by‑component. Most organizations start with Kubeflow Pipelines.

***

# 4. Focus on Kubeflow Pipelines (KFP)

Kubeflow Pipelines are the heart of Kubeflow and provide CI/CD for ML workflows. Each stage of the ML lifecycle (load data, train model, evaluate model, etc.) is treated as a **component**, which runs as a separate **container pod** in the Kubernetes cluster.

Key ideas:

*   ML engineers write a Python script
*   MLOps engineers add KFP DSL (decorators and definitions)
*   KFP SDK compiles Python code into a YAML pipeline
*   YAML is submitted to Kubernetes, which runs it as pods

***

# 5. Installing Kubeflow Pipelines Standalone

Kubeflow Pipelines require a Kubernetes cluster.  
The lecture uses **kind** (Kubernetes in Docker) for local installation.

Steps:

1.  Install Docker Desktop
2.  Install Kind
3.  Create a Kubernetes cluster
4.  Install Kubeflow Pipelines CRDs
5.  Apply platform‑agnostic manifests
6.  Wait for pods to become “Running”
7.  Port‑forward the pipelines UI and access it via `localhost:8080`

Argo Workflows is used under the hood for pipeline execution management.

***

# 6. Writing the First Kubeflow Pipeline (Hello World)

Pipeline creation steps:

1.  Create a Python script
2.  Import KFP libraries: `kfp` and `kfp.dsl`
3.  Convert functions into pipeline components using `@dsl.component`
4.  Define a pipeline orchestrator using `@dsl.pipeline`
5.  Compile to YAML using:
        kfp.compiler.Compiler().compile()
6.  Upload the YAML file to Kubeflow Pipelines UI
7.  Run the pipeline and check the logs

Each run creates Kubernetes pods in the Kubeflow namespace.

This example verifies the installation and establishes core pipeline concepts.

# 7. Real‑World Example: Iris Model Pipeline

The lecture builds a simple ML training pipeline using the Iris dataset.

Three important parts:

### 1. Data Loading Component

*   Loads the Iris dataset
*   Uses pandas and scikit‑learn
*   Decorated with `@dsl.component`

### 2. Model Training Component

*   Uses RandomForestClassifier
*   80% of the data for training
*   20% for validation
*   Logs accuracy
*   Decorated with `@dsl.component`

### 3. Pipeline Orchestration

*   Pipeline combines both components
*   Uses `@dsl.pipeline`
*   Compiled to YAML and uploaded to UI

Execution creates two pods:

*   One to load data
*   One to train the model

Kubeflow displays logs and the final accuracy (example: 0.9).

If the ML team updates their dataset or training logic, they simply re‑run the pipeline and see how the model’s accuracy changes. This eliminates repeated manual execution of each step.

***

# 8. What Kubeflow Pipelines Achieve

Kubeflow Pipelines automate ML workflows:

*   Each run is repeatable
*   Each stage is isolated in a container
*   Stages can scale independently
*   ML engineers no longer manually run scripts step by step
*   Accuracy monitoring becomes consistent
*   Retraining becomes easy
*   Teams can track and compare pipeline runs over time

This drastically reduces operational overhead and speeds up model development.

***

# 9. Key Takeaways

*   Kubeflow is a Kubernetes‑native MLOps platform.
*   KFP is the most important and widely used component.
*   Pipelines are written in Python and compiled into YAML.
*   Each ML stage becomes a pipeline component (containerized execution).
*   Kubeflow is highly extensible and modular.
*   Installing only Kubeflow Pipelines is often enough for many teams.
*   Real ML workflows become manageable, repeatable, and production‑ready.
