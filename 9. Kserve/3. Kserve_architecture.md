# KServe Architecture

*Complete lecture note including all transcript content*

## 1. Introduction

In the previous lecture, we learned the fundamentals of **KServe** and why it is an essential platform for MLOps engineers.  
In this lecture, we focus on the **architecture of KServe**, which is crucial to understand before installing or using it. Once you understand how KServe works internally, implementing and extending it becomes significantly easier.

This explanation uses a practical example to help you remember the KServe architecture permanently.

***

## 2. The Real-World Example: Netflix Recommendation System

Consider a company like **Netflix**.

*   Netflix uses multiple machine learning (ML) models.
*   One of the most important is the **Recommendation Engine**.
*   When a user logs in, Netflix shows recommended movies and series — all powered by ML models.

Netflix wants:

*   Faster model deployment,
*   Automated model serving,
*   Easy inference handling.

To achieve this, Netflix decides to adopt **KServe**.

The MLOps team now begins preparing the environment needed for KServe.

***

## 3. Step 1: Kubernetes Cluster Setup

KServe runs **on top of Kubernetes**, so the first requirement is a Kubernetes cluster.

Two possibilities:

1.  MLOps engineers create a new Kubernetes cluster
2.  They reuse an existing cluster shared with DevOps or other platform teams

Often, KServe is deployed into a cluster shared by multiple organizational teams.

***

## 4. Step 2: Create a Namespace for KServe

MLOps engineers create a dedicated namespace such as:

    kserve-system

This namespace is similar to how DevOps teams install and manage:

*   Argo CD controller
*   Istio controller
*   Prometheus
*   Cert-manager

KServe, like these tools, is installed as a **controller** inside its namespace.

***

## 5. Step 3: Deploy the KServe Controller

The MLOps engineers deploy the **KServe Controller** into the KServe namespace.

This is done using:

*   Helm charts (recommended)
*   Or KServe manifest files (YAML)

### What is the KServe Controller?

*   It is a **cluster-scoped controller**
*   It watches all namespaces in the cluster
*   It monitors KServe custom resources created by different teams
*   It automates the model deployment process

Once installed, the controller continuously watches the cluster for a resource called **InferenceService**.

***

## 6. Multiple Teams, Multiple Namespaces

A Kubernetes cluster often serves multiple teams. For example:

*   Namespace `recommendation-service` for Netflix recommendation models
*   Namespace `payments` for fraud detection or payment models
*   Namespace `user-behavior` for engagement predictions
*   Namespace `ads` for advertising models

Each namespace may have its own MLOps engineer.

KServe can serve *all* teams within the same cluster using **InferenceService CRDs**, without the teams interfering with each other.

***

## 7. Step 4: Creating the InferenceService Resource

A dedicated MLOps engineer working for the **Recommendation Team** now creates an **InferenceService** YAML file.

### What is an InferenceService?

An `InferenceService` is a **Custom Resource Definition (CRD)** created by the KServe project.

This CRD allows MLOps engineers to specify:

*   Where the model file is stored
*   Which ML framework it belongs to
*   How to serve the model
*   Autoscaling behavior
*   Whether to use Ingress or Gateway API

The only requirement for the engineer is:

*   Provide the **model location** (S3, GCS, Azure Blob, MinIO, etc.)

Example:

    s3://netflix-models/recommendation-model/v3/model.pkl

Once the InferenceService is created, the KServe controller begins its automation.

***

## 8. Step 5: What the KServe Controller Does Internally

As soon as the InferenceService is applied:

### 8.1 The KServe Controller detects the new resource

It watches all namespaces and sees:

    InferenceService created in recommendation-service namespace

### 8.2 KServe reads the model location

It extracts the following metadata:

*   Storage bucket path
*   Framework type (scikit-learn, TensorFlow, PyTorch, XGBoost, etc.)
*   Runtime requirements

### 8.3 KServe automatically creates a Deployment with HPA

KServe creates:

*   A **Deployment**
*   With a **Horizontal Pod Autoscaler (HPA)**
*   The Deployment then creates **Pods**

These Pods run a pre-packaged **model server container**.  
KServe ships with runtime servers for multiple ML frameworks.

The pods:

*   Download the model
*   Load the model
*   Start an inference server automatically

### 8.4 KServe creates a Service

A Kubernetes **Service** is automatically created to:

*   Expose the Pods
*   Provide stable service discovery
*   Allow internal routing

### 8.5 KServe creates Ingress or Gateway resources

Depending on installation options:

*   KServe creates an **Ingress resource**
*   Or a **Gateway API resource**

This determines how external users can reach the model.

### 8.6 Users can now access the model

Final traffic flow:

    User → Load Balancer → Ingress / Gateway → Service → Pod → Model Container

The model server handles inference and sends the response back to the user.

***

## 9. Extensibility: Ingress vs Gateway API

While installing KServe using Helm, engineers can choose one of two ways to expose the model:

1.  **Ingress-based routing**
2.  **Gateway API-based routing**

Both options are natively supported by KServe.

A parameter in the installation configuration decides which one is used.

***

## 10. Framework Independence

One of KServe’s biggest strengths:

*   It supports multiple ML frameworks
*   It automatically selects runtime servers
*   It eliminates the need for custom API code or manual serving logic

Supported frameworks include:

*   scikit-learn
*   TensorFlow
*   PyTorch
*   XGBoost
*   ONNX
*   LightGBM
*   Custom Python models via KServe’s Python runtime

Thus, an MLOps engineer does not need to know the intricacies of each framework.

***

## 11. Summary of the KServe Architecture Workflow

1.  Create or reuse a Kubernetes cluster
2.  Create a namespace for KServe
3.  Deploy the KServe controller (Helm chart)
4.  Teams create **InferenceService** YAMLs
5.  KServe monitors all namespaces
6.  KServe reads the model location
7.  KServe automatically creates:
    *   Deployments
    *   Autoscalers
    *   Pods
    *   Services
    *   Ingress/Gateway
8.  End users access the model through a public URL
9.  Requests route to the Pod running the model container
10. Model performs inference and returns the response

This is the complete KServe architecture.

***

## 12. Conclusion

KServe is a Kubernetes-native, fully automated framework that dramatically simplifies:

*   Model deployment
*   Model serving
*   Model inference
*   Autoscaling
*   Framework compatibility
*   Traffic routing
*   Production rollout

With KServe, what used to take days of work can now be completed in **a few YAML files**.
