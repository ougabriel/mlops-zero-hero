# KServe End-to-End Demo for the Intent Classifier Model

*Complete lecture note including all transcript content*

## 1. Overview

In the previous lecture, we deployed a **sample model** using KServe.  
In this lecture, we perform an **end-to-end KServe deployment** for **our actual Intent Classifier model**.

To make your experience smooth, a dedicated **kserve branch** has been added to the GitHub repository.  
This branch contains:

*   `implementation.md`
*   All commands used in this lecture
*   Templates for the InferenceService resource

This demo starts **from scratch**, covering:

1.  Creating a Kubernetes cluster
2.  Installing KServe dependencies
3.  Installing KServe controller
4.  Preparing the model (.pkl)
5.  Hosting the model via GitHub Releases
6.  Creating an InferenceService
7.  Running inference on the serving endpoint

Total time required: \~10 minutes.

***

## 2. Step 1 — Create a Kubernetes Cluster (kind)

We begin by creating a fresh local Kubernetes cluster:

```bash
kind create cluster --name kserve-demo-intent
```

You may also use:

*   AWS EKS
*   Google GKE
*   Azure AKS
*   OpenShift

But **kind** is suitable for this demonstration and avoids cloud costs.

***

## 3. Step 2 — Install cert-manager

KServe requires **cert-manager** because real-time inference typically uses HTTPS (TLS).  
Instead of creating and rotating certificates manually, cert-manager manages them.

Install cert-manager:

```bash
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.12.0/cert-manager.yaml
```

Verify all pods are running:

```bash
kubectl get pods -n cert-manager
```

Do NOT continue until all pods show `Running`.

***

## 4. Step 3 — Install KServe CRDs

Create a namespace for KServe:

```bash
kubectl create namespace kserve
```

Install KServe CRDs:

```bash
kubectl apply -f https://github.com/kserve/kserve/releases/download/v0.11.2/kserve-crds.yaml -n kserve
```

CRDs must be installed **before** the controller, because the controller depends on them to watch custom resources such as `InferenceService`.

***

## 5. Step 4 — Install the KServe Controller

Install using Helm:

```bash
helm install kserve kserve/kserve \
  --namespace kserve
```

This installs:

*   KServe controller
*   KNative Serving components
*   Supporting components required for inference

Give the controller 1–2 minutes to initialize:

```bash
kubectl get pods -n kserve
```

***

## 6. Step 5 — Prepare the Intent Classifier Model for Hosting

KServe needs access to the model file (`intent_model.pkl`).

Instead of using S3 or GCS, we follow an approach many real organizations prefer:

### Store the model as a **GitHub Release artifact**.

Why?

*   Models and source code stay versioned together
*   CI/CD pipelines can upload model artifacts automatically
*   KServe can download binaries from GitHub releases via HTTPS

### Generate the model file (if needed)

If you do not already have `intent_model.pkl`:

```bash
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install -r requirements.txt
python3 model/train.py
```

The generated model file will be available at:

    model/artifacts/intent_model.pkl

### Upload as a Release Asset

1.  Go to **GitHub → Releases**
2.  Click **Draft a new release**
3.  Tag: `v3.0`
4.  Title: `v3.0`
5.  Upload `intent_model.pkl` under “Attach binaries”
6.  Click **Publish Release**

Copy the file’s direct download link.

***

## 7. Step 6 — Create Namespace for the Intent Classifier Service

```bash
kubectl create namespace intent
```

***

## 8. Step 7 — Create the InferenceService for the Intent Classifier Model

In the kserve branch, we have a template:

    inference-service.yaml

Update the `storageUri` field to point to the GitHub Release URL:

```yaml
storageUri: "https://github.com/<username>/<repo>/releases/download/v3.0/intent_model.pkl"
```

Apply the InferenceService:

```bash
kubectl apply -f inference-service.yaml -n intent
```

***

## 9. Step 8 — Observe KServe Controller Logs

Check if the controller detects the new InferenceService:

1.  Get the KServe controller pod:

```bash
kubectl get pods -n kserve
```

2.  Tail logs:

```bash
kubectl logs -f <kserve-controller-pod> -n kserve
```

Expected log output:

    Reconciling InferenceService "intent-classifier"
    Creating Deployment...
    Creating Service...
    Creating HorizontalPodAutoscaler...

This confirms that KServe:

*   Read the CRD
*   Created Deployment
*   Created Service
*   Created HPA
*   Loaded the model server runtime automatically

***

## 10. Step 9 — Validate Deployment Components

Check HPA:

```bash
kubectl get hpa -n intent
```

Check Deployment:

```bash
kubectl get deployments -n intent
```

Check Service:

```bash
kubectl get svc -n intent
```

Everything should be created automatically.

***

## 11. Step 10 — Expose the Model using Port Forwarding (Local Demo)

Because we are using a local cluster, we cannot use an external load balancer or Ingress.

Use port-forwarding instead:

```bash
kubectl port-forward svc/intent-classifier-predictor \
  -n intent 8080:80 --address 0.0.0.0
```

If port 8080 is in use, choose another, such as 8089.

***

## 12. Step 11 — Run Inference on the Intents Model

Open another terminal and run:

```bash
curl -X POST "http://localhost:8080/v1/models/intent-classifier:predict" \
  -H "Content-Type: application/json" \
  -d '{"text": "I want to cancel my subscription"}'
```

Example output:

    {"intent": "cancel"}

Try another example:

```bash
curl -X POST "http://localhost:8080/v1/models/intent-classifier:predict" \
  -H "Content-Type: application/json" \
  -d '{"text": "hello"}'
```

Expected output:

    {"intent": "greeting"}

This confirms:

*   Model loaded successfully
*   KServe served the model
*   Prediction API is working

## 13. Summary

With KServe, deploying the Intent Classifier model required only:

1.  Creating the cluster
2.  Installing cert-manager
3.  Installing KServe CRDs
4.  Installing the KServe controller
5.  Uploading the model file
6.  Creating an InferenceService CRD

This dramatically reduces the workload for MLOps engineers and mirrors how real organizations productionize ML models.

You are strongly encouraged to reproduce this demo end‑to‑end to fully internalize the workflow.
