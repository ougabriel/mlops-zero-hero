# KServe End-to-End Demo for a Sample Model

Reference Link: https://github.com/iam-veeramalla/Intent-classifier-model/blob/kserve/KServe-implementation.md

## 1. Overview

This lecture walks through a complete end‑to‑end demonstration of **installing KServe**, configuring its dependencies, deploying a sample model using an **InferenceService**, and performing model inference.

To reduce cloud costs, this demo uses a **local Kubernetes cluster** created with `kind`.  
You must have the following installed:

*   Docker
*   kind
*   kubectl
*   Helm

These tools were installed in previous sections, so this lecture focuses directly on the demo.

***

## 2. Step 1 — Create a Local Kubernetes Cluster (kind)

Start by creating a local cluster:

```bash
kind create cluster --name mlops-kserve-demo
```

Cluster creation typically takes 60–120 seconds.

### Verify kubectl context

Always confirm that kubectl is pointing to the correct cluster:

```bash
kubectl config current-context
```

This step prevents accidental deployment onto a production or unrelated cluster.

***

## 3. Step 2 — Install cert-manager (Required by KServe)

KServe requires **cert-manager**, because cert-manager manages certificates used for secure HTTPS communication (TLS) in real-time environments.

Install cert-manager:

```bash
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.12.0/cert-manager.yaml
```

### Important:

Wait until **all cert-manager pods are in Running state**:

```bash
kubectl get pods -n cert-manager
```

Do not proceed until they are fully ready.

***

## 4. Step 3 — Install KServe CRDs

Before installing the KServe controller, you must install the KServe **Custom Resource Definitions (CRDs)**.

First, create the namespace:

```bash
kubectl create namespace kserve
```

Install CRDs (version may vary):

```bash
kubectl apply -f https://github.com/kserve/kserve/releases/download/v0.11.2/kserve-crds.yaml -n kserve
```

CRDs define KServe resources such as:

*   `InferenceService`
*   `TrainedModel`
*   `ClusterServingRuntime`
*   `ServingRuntime`

***

## 5. Step 4 — Install KServe Controller

Install KServe using Helm:

```bash
helm install kserve kserve/kserve \
  --namespace kserve
```

This installs the KServe controller, which:

*   Watches all namespaces
*   Detects new `InferenceService` objects
*   Automatically creates deployments, services, autoscalers, networking, etc.

It may take up to 2 minutes for all KServe pods to become ready:

```bash
kubectl get pods -n kserve
```

***

## 6. Step 5 — Create a Namespace for the Model

Create a namespace for the ML team:

```bash
kubectl create namespace ml
```

This is where we deploy our model.

***

## 7. Step 6 — Deploy an InferenceService (Sample Model)

KServe provides many sample models stored publicly.  
In this demo, we use the **Sklearn Iris** model hosted in a Google Cloud Storage bucket:

Example InferenceService YAML:

```yaml
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: sklearn-iris
  namespace: ml
spec:
  predictor:
    sklearn:
      storageUri: gs://kfserving-examples/models/sklearn/iris
```

Apply it:

```bash
kubectl apply -f sklearn-inference.yaml
```

***

## 8. Step 7 — Observe KServe Controller Behavior

Check KServe controller logs:

```bash
kubectl get pods -n kserve
kubectl logs -f <kserve-controller-pod> -n kserve
```

You will see messages such as:

    Reconciling InferenceService sklearn-iris
    Creating Deployment...
    Creating Service...
    Creating HorizontalPodAutoscaler...

These logs confirm:

*   KServe detected the new InferenceService
*   It created Deployment + HPA
*   It created a Service
*   The model server pod has started

***

## 9. Step 8 — Verify Resources in the `ml` Namespace

Check the Service:

```bash
kubectl get svc -n ml
```

You should see something like:

    sklearn-iris-predictor

Check pods:

```bash
kubectl get pods -n ml
```

Pods should be running.

***

## 10. Step 9 — Expose the Model (Local Port Forwarding)

Because this demo uses a **local cluster**, we cannot use an external load balancer or Ingress.  
We therefore use **port-forwarding** to expose the model locally.

Run:

```bash
kubectl port-forward svc/sklearn-iris-predictor -n ml 8089:80 --address 0.0.0.0
```

If port 8089 is busy, choose another one.

***

## 11. Step 10 — Run Inference with curl

Open a new terminal and send an inference request:

```bash
curl -s -X POST "http://localhost:8089/v1/models/sklearn-iris:predict" \
  -H "Content-Type: application/json" \
  -d '{"instances": [[5.1, 3.5, 1.4, 0.2]]}'
```

Expected output:

    {"predictions": [2]}

This means:

*   The Iris model successfully loaded
*   The request reached the pod
*   The model server responded with prediction “2”

KServe has now fully deployed and served the model.

***

## 12. Summary

Once initial setup is complete (cluster, cert-manager, KServe controller), MLOps engineers only need to:

1.  Create a namespace
2.  Provide the model’s storage URI
3.  Apply a single `InferenceService` YAML

KServe automatically:

*   Loads the model
*   Creates deployments
*   Configures networking
*   Sets up autoscaling
*   Exposes the model
*   Manages revisions
*   Provides standardized inference APIs

This drastically simplifies model deployment and serving.
