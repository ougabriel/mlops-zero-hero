# Introduction to KServe

*Complete lecture note including all transcript content*

## 1. Overview

This lecture introduces **KServe**, a critical open‑source platform widely used by MLOps engineers for **model deployment, model serving, and model inference**.  
KServe is fully **Kubernetes-native**, and is designed to automate and simplify complex model operationalization workflows.

Before learning KServe, it’s important to revisit the **machine learning lifecycle**, because KServe primarily solves challenges that arise after a model is trained.

***

## 2. Machine Learning Lifecycle Recap

### 2.1 Data Scientists’ Responsibilities

Data scientists typically:

1.  Receive business requirements
2.  Collect or prepare a dataset
3.  Select or experiment with ML algorithms
4.  Train the model
5.  Save the trained model in formats such as:
    *   `.pkl` (scikit-learn)
    *   `.joblib`
    *   TensorFlow SavedModel format
    *   PyTorch `.pt`
    *   XGBoost model files

At this stage, the model *exists*, but it is **not yet usable by end users**.  
The job is only **50% complete**.

### 2.2 MLOps Engineers’ Responsibilities

After the model is created, MLOps engineers must:

*   Deploy the model
*   Create a serving API
*   Manage inference requests from end users
*   Handle scaling
*   Manage infrastructure, networking, load balancing, security, and reliability

These activities traditionally require **significant manual effort**.

***

## 3. The Problem with Manual Deployment Approaches

In previous sections, we deployed models using:

*   Virtual Machines
*   Kubernetes with Deployments, Services, and Ingress

Although both approaches work, they involve many manual, time-consuming steps.

### 3.1 Challenges in VM-Based Deployment

When deploying to VMs, engineers must:

*   Write a Flask API
*   Configure WSGI (Gunicorn)
*   Configure NGINX as reverse proxy
*   Provision VM infrastructure
*   Configure autoscaling
*   Configure load balancers
*   Manage networking and firewalls

This can take **several days** even for small models.

### 3.2 Challenges in Kubernetes-Based Deployment

In the Kubernetes section, engineers still had to:

*   Create Flask API
*   Configure WSGI
*   Build Docker containers
*   Write Deployment YAML
*   Write Service YAML
*   Implement HPA or KEDA for autoscaling
*   Configure Ingress
*   Manage networking and routing

Again, a large set of manual steps.

***

## 4. What is KServe?

**KServe** is an **open-source Kubernetes-native platform** that automates:

*   Model deployment
*   Model serving
*   Model inference

KServe reduces operational burden by handling all model-serving complexities internally.

Instead of writing APIs, containers, manifests, and autoscaling logic, MLOps engineers simply:

1.  Upload the model to a storage location (S3, GCS, MinIO, etc.)
2.  Provide the model path to a KServe configuration
3.  Apply a single custom resource definition (CRD) YAML

KServe automatically performs everything else.

***

## 5. Why Use KServe?

### 5.1 Automates Manual Tasks

KServe eliminates many tasks that otherwise require manual DevOps work:

*   API creation
*   WSGI setup
*   NGINX reverse proxy configuration
*   Docker image creation
*   Deployment and Service creation
*   Autoscaling setup
*   Load balancer configuration
*   Traffic routing via Knative or Istio

What previously took days may now take **minutes**.

### 5.2 Supports Multiple ML Frameworks

This is one of the strongest advantages of KServe.

Data scientists may use different ML frameworks:

*   scikit-learn
*   TensorFlow
*   XGBoost
*   PyTorch
*   ONNX
*   LightGBM
*   Custom Python models

With KServe:

*   MLOps engineers do *not* need to learn separate serving stacks for each framework
*   KServe automatically knows how to load the model and expose a consistent inference API

This drastically reduces the learning curve.

### 5.3 Built-In Autoscaling (Knative + KEDA)

Traditional Kubernetes requires manual setup of:

*   Horizontal Pod Autoscaler (HPA)
*   Vertical Pod Autoscaler (VPA)
*   KEDA
*   Knative Serving

KServe integrates with:

*   **Knative** for serverless autoscaling (scale-to-zero support)
*   **KEDA** for event-driven scaling

This means:

*   If no requests are coming, pods can scale to **zero**
*   If heavy traffic arrives, KServe automatically scales pods up to 10, 20, or more
*   Autoscaling rules are handled internally with minimal configuration

### 5.4 Kubernetes Native CRDs

KServe extends Kubernetes using **Custom Resource Definitions (CRDs)**.

Engineers deploy a model using a simple YAML such as:

    InferenceService:
      ...

KServe takes care of:

*   Routing
*   Pods
*   Revisions
*   Canary rollouts
*   Logging
*   Monitoring

All with minimal complexity.

***

## 6. Summary

KServe is a Kubernetes-based, open-source model serving platform that significantly simplifies the MLOps workflow.

### Key Benefits of KServe:

*   Automates model deployment, serving, and inference
*   Works with multiple ML frameworks (scikit-learn, TF, XGBoost, PyTorch, etc.)
*   Reduces manual workload (no need for Flask, Gunicorn, NGINX, Docker, Kubernetes YAMLs)
*   Provides built-in autoscaling using Knative and KEDA
*   Supports serverless scale-to-zero
*   Makes model deployment faster, easier, and more efficient
